# AGENTS — Dotfiles (Omarchy family)

Hard rules for an agent (human or AI) to work in this repo without breaking conventions. If a rule here conflicts with an installed skill, this file wins. Read it fully before your first edit: each section states its scope and limits.

## Stack and ownership

Real repo stack (don't invent domains):

| Path                                                  | Lives at                       | Owner                 | Agent may                                              |
| ----------------------------------------------------- | ------------------------------ | --------------------- | ------------------------------------------------------ |
| `src/home/<name>`                                     | `~/.<name>`                    | this repo             | edit freely                                            |
| `src/home/config/<app>/`                              | `~/.config/<app>/`             | this repo             | edit; survives `omarchy update`                        |
| `src/home/local/bin/<name>`                           | `~/.local/bin/<name>`          | this repo             | edit; personal and manual                              |
| `src/home/ssh/config`                                 | `~/.ssh/config`                | this repo             | edit; rest of `~/.ssh/` excluded from repo             |
| `src/etc/<path>`                                      | `/etc/<path>` (root-installed) | this repo (carefully) | edit; requires root on install                         |
| `src/utils/bash/<name>`                               | invoked by `./setup`           | this repo             | edit; affects the entrypoint and the env-var contract  |
| `docs/`, `setup`, `README.md`, `AGENTS.md`            | repo root                      | this repo             | edit freely                                            |
| `backup/`                                             | runtime, gitignored            | generated by `./setup` | no commit content; ignore in diffs                     |
| Paths in `~/` or `~/.config/` NOT under `src/`        | live filesystem                | Omarchy installer     | no regenerate; ask before touching                     |

Practical rule: if a path doesn't appear in the table and isn't under `src/`, `docs/`, or root, it belongs to Omarchy. Don't edit it without explicit confirmation.

Naming convention: source paths under `src/home/` carry no leading `.` (e.g., `src/home/bashrc`, `src/home/ssh/config`); the symlinked or copied target at `$HOME` adds the leading `.` for hidden configs (`~/.bashrc`, `~/.ssh/config`).

## Layout

```
setup                       # entrypoint root: single bash dispatcher
src/home/                   # symlinks to ~/
src/etc/                    # symlinks to /etc/
src/utils/bash/             # helpers invoked by ./setup
docs/                       # specific guides (hypr.md, nvim/, setup.md)
backup/                     # GENERATED by ./setup, gitignored
```

Detail in `README.md` (section Repo Layout). The current `setup` is the style reference for any new bash in the repo.

## User communication

- Short, direct responses by default.
- Respond in Spanish Neutral Technical, without regional variants or jargon.
- One question at a time. After asking, STOP and wait.

## Commits and pushes

- **Commit**: only if the user asks explicitly.
- **Push**: only if the user asks explicitly.
- Messages in Conventional Commits. No "Co-Authored-By" or AI attribution. No emojis.

## Inline comments and documentation

- **Language**: English (US), technical, regardless of conversation language.
- **What to document**: exports (functions, classes, types, interfaces) when the contract isn't obvious. Skip self-explanatory helpers and one-liners.
- **Content**: why (intent, decision, gotcha), not what.
- **Commented-out code**: forbidden. Delete it; git preserves history.
- **Emojis**: forbidden in code and commits.

## Verification of technical claims

- Don't assume APIs, conventions, or behavior from memory. Verify against official docs before writing or modifying code.
- Cite the source (docs URL + package version) on non-obvious technical claims.
- If the user marks something as incorrect: verify against the docs before accepting or rejecting. Memory and "probably" are not evidence.

## User changes to generated code

- **Assume intent**: any difference between what you generated and what's in the repo is, by default, intentional.
- **No revert without confirmation**: don't undo, rewrite, or "fix" those changes without explicit confirmation.
- **How to ask**: if you consider it an error or bug, raise the observation with evidence (URL, line, diff) and ask before touching.
- **Exception**: if the user explicitly asked to revert or adjust ("go back", "apply this instead of the previous one"), proceed.
- **Out of scope**: if your diff touches `backup/` (generated by setup), `src/home/ssh/*` except `config`, or paths outside `src/`, `docs/`, and root, it's noise. Filter it from the report.

## Sub-delegation

This repo is small. Don't delegate to sub-agents to edit a `.conf` or a 50-line script. Delegation is justified only if both conditions hold:

- The change crosses 4+ non-trivial files.
- You need to read 4+ files to understand the context before writing.

If in doubt, work inline.
