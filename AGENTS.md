# AGENTS — Dotfiles (Omarchy family)

Hard rules for an agent (human or AI) to work in this repo without breaking
conventions. If a rule here conflicts with an installed skill, this file wins.
Read it fully before your first edit: each section states its scope and limits.

## Layout

The repo owns `src/`, `docs/`, and the root (`setup`, `README.md`, `AGENTS.md`); everything
there is free to edit. Caveats: `src/home/config/<app>/` survives `omarchy update`;
`src/home/local/bin/` is personal and manual; `src/home/ssh/` is limited to `config` (the rest
of `~/.ssh/` is excluded from the repo); `src/etc/` requires root on install — edit carefully;
`src/utils/bash/` is invoked by `./setup` and affects the entrypoint and the env-var contract.
`backup/` is generated by `./setup` and gitignored — never commit its content. Anything under
`~/` or `~/.config/` not symlinked from `src/` belongs to the Omarchy installer; don't
regenerate or edit it without explicit confirmation.

Naming convention: source paths under `src/home/` carry no leading `.`
(e.g., `src/home/bashrc`, `src/home/ssh/config`); the symlinked or copied target
at `$HOME` adds the leading `.` for hidden configs (`~/.bashrc`, `~/.ssh/config`).

```
setup                       # entrypoint root: single bash dispatcher
src/home/                   # symlinks to ~/
src/etc/                    # symlinks to /etc/
src/utils/bash/             # helpers invoked by ./setup
docs/                       # specific guides (hypr.md, nvim/, setup.md)
backup/                     # GENERATED by ./setup, gitignored
```

Detail in `README.md` (section Repo Layout). The current `setup` is the style reference
for any new bash in the repo.

## User communication

- Short, direct responses by default.
- Respond in Spanish Neutral Technical, without regional variants or jargon.
- One question at a time. After asking, STOP and wait.

## Commits and pushes

- **Commit**: only if the user asks explicitly.
- **Push**: only if the user asks explicitly.
- Messages in Conventional Commits. No "Co-Authored-By" or AI attribution. No emojis.

## Inline comments and documentation

- **Language**: English (US), technical, regardless of conversation language.
- **What to document**: exports (functions, classes, types, interfaces) when the contract
  isn't obvious. Skip self-explanatory helpers and one-liners.
- **Content**: why (intent, decision, gotcha), not what.
- **Commented-out code**: forbidden. Delete it; git preserves history.
- **Emojis**: forbidden in code and commits.

## Verification of technical claims

- Don't assume APIs, conventions, or behavior from memory. Verify against official
  docs before writing or modifying code.
- Cite the source (docs URL + package version) on non-obvious technical claims.
- If the user marks something as incorrect: verify against the docs before accepting or
  rejecting. Memory and "probably" are not evidence.

## User changes to generated code

- **Assume intent**: any difference between what you generated and what's in the
  repo is, by default, intentional.
- **No revert without confirmation**: don't undo, rewrite, or "fix" those changes
  without explicit confirmation.
- **How to ask**: if you consider it an error or bug, raise the observation with evidence
  (URL, line, diff) and ask before touching.
- **Exception**: if the user explicitly asked to revert or adjust ("go back",
  "apply this instead of the previous one"), proceed.
- **Out of scope**: if your diff touches `backup/` (generated by setup), `src/home/ssh/*`
  except `config`, or paths outside `src/`, `docs/`, and root, it's noise.
  Filter it from the report.

## Sub-delegation

This repo is small. Don't delegate to sub-agents to edit a `.conf` or a 50-line script.
Delegation is justified only if both conditions hold:

- The change crosses 4+ non-trivial files.
- You need to read 4+ files to understand the context before writing.

If in doubt, work inline.
